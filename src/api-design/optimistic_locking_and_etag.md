## 概要

楽観的排他制御（Optimistic Locking）は、リソースの更新や削除を行う際に、複数クライアントからの同時更新による「更新の喪失（Lost Update）」を防ぐための手法です。特に、Web APIのようなステートレスな通信においては、データベースの行ロック（悲観的ロック）を長時間維持することがパフォーマンス低下やデッドロックの原因となるため、バージョン番号またはETagを用いてバージョンの一致を確認することが重要です。

## なぜ必要なのか

複数のユーザーやデバイスが同時に同じリソースを取得し、それぞれが更新を行った場合、後から保存されたデータによって前の更新が上書きされ、消失する可能性があります。このような状況を防ぐために、楽観的排他制御が必要です。

## 実装標準

1. **DB設計:** 対象テーブルに `version` という数値カラムを追加する。
2. **参照時 (GET):** データを返す際、現在のバージョン番号をレスポンスヘッダの `ETag` またはJSONボディ内に含める。
3. **更新時 (PUT/PATCH):** クライアントは更新リクエスト時に、取得時に受け取ったバージョンを `If-Match` ヘッダまたはJSONボディに含めて送信する。
4. **検証と保存:** サーバー側は、受け取ったバージョンとDBの現在のバージョンを比較し、一致していればデータを更新し、バージョンをインクリメントする。不一致の場合は、HTTPステータス `412 Precondition Failed` または `409 Conflict` を返す。

## 設計・実装時のチェックリスト

- [ ] 複数人が編集する可能性のあるリソースのAPI設計において、バージョン管理と排他制御が組み込まれているか？
- [ ] DB更新時に、必ず `version` カラムを用いた条件指定（WHERE句）を行っているか？
- [ ] 競合エラー（409/412）が発生した際、クライアントが適切にユーザーに通知できるエラーレスポンスになっているか？

## アンチパターン

- **Last-Write-Wins（後勝ち）の放置:** 排他制御を実装せず、最後に届いたリクエストでデータを上書きしてしまう。
- **Web APIでの悲観的ロック:** ユーザーが編集画面を開いている間、DBのロックを保持し続ける設計。これにより、他のユーザーの操作がブロックされ、システム全体が停止する原因となる。

## 実施・導入によるトレードオフ

- フロントエンドやモバイルアプリ側で、競合エラーを受け取った際のリロードやユーザーへのメッセージ表示など、エラーハンドリングの実装コストが増加する。
- DBのテーブル設計において、バージョンカラムの追加とメンテナンスが必須となる。

## 適用範囲と例外

- **スコープ:** 商品情報、ユーザー情報、グループ設定など、複数人や複数プロセスから更新される可能性のあるすべての「集約」の更新・削除API。
- **例外:** 一人のユーザーしかアクセスせず、単一のデバイスからしか操作されないことが保証されている一時データや、単なるログの追記（INSERTのみの処理）。

## 検証方法

統合テストにて、同じデータに対して2つの非同期リクエストを同時に送信し、片方が正常終了（HTTP 200）し、もう一方が必ず競合エラー（HTTP 412 または 409）になることを確認する。
